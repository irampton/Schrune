# CLI

Schrune installs a `schrune` command from `package.json`.

## Install

```bash
npm install
npm link
```

After linking, `schrune` is available on your `PATH`.

If you plan to import LCSC parts, install the Python converter into the active
Python environment first:

```bash
npm run setup:easyeda2kicad
```

If you need a specific interpreter, set `EASYEDA2KICAD_PYTHON` before running
the script.

## Commands

### `schrune create`

Create a `schrune.json` project file. The CLI prompts for a project name and
entry `.schrune` file, then writes the manifest next to that entry file.

### `schrune build [-l|--update-layout] [file.schrune]`

Compile a Schrune source file, fetch or resolve parts, and generate KiCad output.
When no file is passed, Schrune resolves the nearest `schrune.json` and uses its
configured entry file.

Example:

```bash
schrune build example3/example3.schrune
```

Output:

* An intermediate JavaScript file when `--keep-js` is used
* A BOM CSV in `build/`
* A `build/` directory containing the project, schematic, and PCB files
* Updated `build` paths recorded into `schrune.json` when a project manifest is used

By default, builds do not touch an existing `build/<name>.kicad_pcb`. If the PCB
file does not exist, Schrune creates an empty board file linked from the KiCad
project so KiCad can update it from the generated schematic. Use
`-l`/`--update-layout` to refresh/populate the PCB layout during the build.

### `schrune open-kicad`

Open the generated KiCad project recorded in `schrune.json` using the platform
default application launcher.

### `schrune parts add <CXXXX>`

Download a part library entry for an LCSC part number. This requires the Python
`easyeda2kicad` package to be installed in a usable interpreter.

Example:

```bash
schrune parts add C29823
```

This creates a part directory under `parts/` with the Schrune source and KiCad assets for the part.
When a `schrune.json` project file is present, the imported part is also added
to its `parts` array as `{ "MPN": "...", "LCSC": "..." }`.

### `schrune parts install`

Install all parts listed in the `parts` array of `schrune.json`. This command
requires a `schrune.json` in the current directory and does not search parent
directories.

With `--update`, Schrune redownloads the symbol, footprint, and model files and
updates generated `info` fields inside existing `.schrune` files without
replacing custom comments or pin edits.

With `--include-autogenerated`, Schrune also reads
`parts/autogenerated/parts-lock.json` for autogenerated parts and installs or
updates those imports under `parts/autogenerated/`.

### `schrune tools jlc-bom [-n N]`

Read the current project's generated BOM CSV, query JLCPCB for each LCSC part,
and print a terminal BOM grouped into Basic, Promoted Extended, and Extended
sections.

This command requires a `schrune.json` in the current directory and a previously
generated `build/<name>.BOM.csv`.

Without `-n`, the per-board pricing columns assume 5 boards and the output
shows summary assembly estimates. With `-n`, the per-board pricing columns use
that board count, the output includes full economic and standard line-item
breakdowns for that quantity, and it reports potential stock shortages.

## Flags

### `--keep-js`

Write the intermediate Step 1 JavaScript alongside the source file.

### `--no-parts-lock`

Skip reading and writing the autogenerated parts lock file.

### `-l`, `--update-layout`

For `schrune build`, update the generated PCB layout. Without this flag, an
existing `.kicad_pcb` is left untouched and a missing one is created empty.

### `-u`, `--update`

For `schrune parts install`, refresh installed parts in place instead of only
installing missing ones.

### `-a`, `--include-autogenerated`

For `schrune parts install`, include entries from the autogenerated parts lock.

### `-n N`

For `schrune tools jlc-bom`, use `N` boards when pricing component tiers and
assembly estimates.

## Compatibility Mode

You can also invoke the build flow directly:

```bash
schrune <file.schrune>
```

This is treated as `schrune build <file.schrune>`.

## Build Progress

The CLI reports the build in stages:

* Parsing source
* Compiling nets
* Fetching components
* Sending to KiCad

The component fetch stage shows progress in `X/XX` form while parts are being resolved.

## Generated Files

The build writes:

* `build/<name>.BOM.csv`
* `build/<name>.kicad_pro`
* `build/<name>.kicad_sch`
* `build/<name>.kicad_pcb` when it does not already exist, or when
  `-l`/`--update-layout` is used

## Notes

* `@require("Name");` resolves exactly `parts/Name/Name.schrune`.
* `@require(Name from "/path/file.schrune");` loads a named module from an explicit Schrune path.
* `@require({First, Second} from "/path/file.schrune");` loads multiple named modules from one file.
* The CLI expects either a `.schrune` input file or a nearby `schrune.json` for build mode.
